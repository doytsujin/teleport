# Getting Started

Teleport Server Access provides full SSH capabilities out-of-the-box:

1. Teleport's `tsh` combines support for SSH and SCP in one convenient local command-line tool.
2. Each Teleport Node can be configured into SSH mode (Teleport Node) and run as an enhanced SSH server.

The steps below involve:

1. Creating a Teleport Auth Node cluster we'll add a Teleport SSH Node to.
2. SSH into the cluster without external SSH dependencies (using `tsh` as an SSH client).
3. Introspecting the cluster using advanced `tsh` features.

## Prerequisites

- A Linux environment (such as Ubuntu 20.05, CentOS 8.0-1905, or Debian 10) within which we'll create multiple instances.
- A registered Doman Name.
- A Two-Factor Authentication app (such as Authy or Google Authenticator).
- Teleport (=teleport.version=) installed locally (for `tsh`).

<Admonition type="warning" title="Warning">
  The examples below may include the use of the `sudo` keyword to make following each step easier when creating resources from scratch. However, we generally discourage the use of `sudo` in **real-world production environments** per the *Principle of Least Privilege* (POLP). 
</Admonition>

## Step 1/5. Create a cluster

<Admonition type="note" title="Note">
  This guide involves configuring your Teleport Nodes into the *Bastion* pattern so that [only a single Node can be accessed publically](https://goteleport.com/blog/ssh-bastion-host/).
</Admonition>

<Admonition type="tip" title="Tip">
  Multiple Teleport Nodes can be run on the same Linux instance. This Guide details how to launch and connect two Nodes on different devices.
</Admonition>

1. Verify you have a registered domain name. 

  - We'll create one subdomain `tele.example.com` that will be publicly available.

2. Create two new instances of your desired Linux distribution (such as Ubuntu 20.05, CentOS 8.0-1905, or Debian 10).

   i. The first instance will host a public-facing Auth Node `tele.example.com` that will serve as the *Bastion Host* (following the recommended *Bastion*-pattern). As such, leave the following ports open:

   | Port | Service | Description |
   | - | - | - |
   | 3023 | Proxy | SSH port clients connect to. A proxy will forward this connection to port `#3022` on the destination Node. |
   | 3025 | Auth | SSH port used by the Auth Service to serve its API to other Nodes in a cluster. |
   | 3080 | Proxy | HTTPS connection to authenticate `tsh` users and web users into the cluster. The same connection is used to serve a Web UI. |
   | 443 | Browser | Used to access the Teleport Web UI through a browser with HTTPS. |
   | 22 | Cloud Provider | Used to initially access, configure, and provision your cloud instances. We'll configure and launch our instances then demonstrate how to use the `tsh` tool and Teleport in SSH mode thereafter. |

   ii. Add an `A` record in your domain registrar entry for the `tele` subdomain:

   - Map the first IP address to the `tele` subdomain.

   iii. The second instance is configured to host a private SSH Node accessible only indirectly via the *Bastion Host*:

   | Port | Service | Description |
   | - | - | - |
   | 3023 | Proxy | SSH port clients connect to. A proxy will forward this connection to port `#3022` on the destination Node. |
   | 3080 | Proxy | HTTPS connection to authenticate `tsh` users and web users into the cluster. The same connection is used to serve a Web UI. |
   | 22 | Cloud Provider | Used to initially access, configure, and provision your cloud instances. We'll configure and launch our instances then demonstrate how to use the `tsh` tool and Teleport in SSH mode thereafter. |

   iv. For a complete list of ports used by Teleport, read [The Admin Guide](../admin-guide.mdx#ports).

   v. Create the necessary SSH keys and connect from your local machine to each using the following `ssh` command format. For example, on AWS EC2: 
   
   - `aws % ssh -i "example.pem" ubuntu@ecX-X-XX-XXX-XX.us-north-2.compute.amazonaws.com`

4. Configure Teleport on the *Bastion Host*.

   i. Teleport will now automatically acquire a [Let's Encrypt](https://letsencrypt.org/docs/client-options/) ACME-compliant TLS certificate. 
   
   While you can use a third-party ACME client (like Let's Encrypt certbot), the configuration commands below will do the same:

   ```bash
   # Configure Teleport with TLS certs
   sudo teleport configure \
     --acme --acme-email=your_email@example.com \
     --cluster-name=tele.example.com \
     -o file
   ```

   Run the command above on `tele.example.com`.

5. Launch your  *Bastion Host*:

   - Execute: `sudo teleport start`.
   - Now, we'll create a *join token* to add and start the second Node.

1. Now, create a third terminal and connect to `tele.example.com`:

   i. Create a user to access the `tele.example.com` Web UI: 

   - `sudo tctl users add tele-admin --roles=editor,access --logins=root,ubuntu,ec2-user`  

   ii. This will generate an initial login link where you can set a password and set up Two-Factor Authentication for `tele-admin`.

   iii. Create a *join token* to add the second Node to the `tele.example.com` cluster:

   - `sudo tctl tokens add --type=node`  
   - "Node" mode specifies that the Teleport Node will act and join as an SSH server.
   - Save the resultant `token` value to a token-specific directory of your choice on the second Node. For example: `/tmp/token`. 
   - This helps to minimize the direct sharing of tokens even when they are *dynamically* generated.

   iv. You should see:

   ```bash
   # Join cluster
   sudo teleport start \
     --roles=node \
     --token=/tmp/token \
     --ca-pin=sha256:c75f8be5b21a5cc1b6f91a71200f30a9a083dfea4fc47ec1c55143ccce8882e4 \
     --auth-server=172.31.17.124:3025
   ```

2. In your second, *Bastion*, terminal:

   i. Run the command above with the correct `tele.example.com` public proxy address on the machine you wish to connect.

   ii. You should now be able to view both Nodes at: `https://tele.example.com/web/cluster/tele.example.com/nodes` in the Web Interface after logging in with your the user `tele-admin`.

## Step 3/5. SSH into the server

Now, we've got our cluster up and running. Let's see how easy it is to connect to our two Nodes. We can use `tsh` to SSH into the cluster:

1. On your local machine, log in through `tsh`: 

   i. `tsh login --proxy=tele.example.com:443 --user=tele-admin`  
   ii. You'll be prompted to supply the password and One Time Passcode we set up previously.

2. `tele-admin` will now see:

   ```bash
   Profile URL:        https://tele.example.com:443
     Logged in as:       tele-admin
     Cluster:            tele.example.com
     Roles:              access, editor
     Logins:             root, ubuntu, ec2-user
     Kubernetes:         disabled
     Valid until:        2021-04-30 06:39:13 -0500 CDT [valid for 12h0m0s]
     Extensions:         permit-agent-forwarding, permit-port-forwarding, permit-pty
   ```

  - `tele-admin` is now logged into the `tele.example.com` cluster and *Bastion Host* Node via Teleport SSH - no third-party tool required.

3. `tele-admin` can now execute the following to find the cluster's `nodenames`. `nodenames` are used for establishing SSH connections:

   i. `tsh ls`  
   ii. The  *Bastion Host* Node is located on the bottom line below:

   ```bash
   Node Name        Address            Labels                                
   ---------------- ------------------ ------------------------------------- 
   ip-172-31-39-190 172.31.39.190:3022                                       
   ip-172-31-40-99  127.0.0.1:3022     env=example, hostname=ip-172-31-40-99 
   ```

4. Using the command `tsh ssh root@ip-172-31-40-99` locally, `tele-admin` can SSH into the *Bastion Host* Node and perform the following:

   i. Connect to other Nodes in the cluster by repeating the steps above.  
   ii. Traverse the Linux file system.  
   iii. Execute desired commands.  

   - All commands executed by `tele-admin` are recorded and can be replayed in the Teleport Web interface.
   - The `tsh ssh` command allows one to do anything they would if they were to SSH into a server using a third-party tool. Compare the two equivalent commands:

## Step 4/5. Use tsh and the Universal Resource Catalog to introspect the cluster

1. Now, `tele-admin` has the ability to SSH into other Nodes within the cluster, traverse the Linux file system, and execute commands.

   - They have visibility into all resources within the cluster due to their defined and assigned role.
   - They can also quickly view any Node or grouping of Nodes that've been assigned a `label`.

2. Execute `sudo tctl nodes ls` within your *Bastion Host* console which displays the Universal Resource Catalog with all queried resources in one view. Currently, we're displaying only those Nodes that are part of our *Bastion* cluster.  
   
   ```bash
   Nodename         UUID                                 Address            Labels                               
   ---------------- ------------------------------------ ------------------ ------------------------------------ 
   ip-172-31-40-99  bf840c02-68f4-4fac-a9e6-30141e82d242 127.0.0.1:3022     env=example,hostname=ip-172-31-40-99 
   ip-172-31-39-190 d32945f4-6e35-4318-b8f6-198d7c1fbd4c 172.31.39.190:3022                                      
   ```

3. Note the "Labels" column on the farthest side. `tele-admin` can query all resources with a shared [label](../admin-guide.mdx#labeling-nodes-and-applications) using the command `tsh ssh root@env=example`.

   - Customized labels can be defined in your `teleport.yaml` configuration file or during Node creation.
   - This is a convenient feature that allows for more advanced queries. 
   - Suppose an IP address changes, an admin can quickly find the current Node with that label since it remains unchanged.

4. `tele-admin` can also execute commands on all Nodes that share a label vastly simplifying repeated operations:

   - `tsh ssh root@env=example ls` will execute the `ls` command on each Node displaying the contents of each to the screen.

## Step 5/5. Connect to the Teleport API

Let's connect to the Teleport API using a simple role-based authentication method and Go client:

1. Open a terminal to `tele.example.com`.

2. Create a user `api-admin` with the built-in role `admin`: `sudo tctl users add api-user --roles=admin`.

3. Within the same terminal log in as `api-admin`: `tsh login --user=api-user --proxy=tele.example.com:443`. This will save and load `api-admin`'s `tsh` profile automatically within the Go client.

4. Install `Go 1.16+` (`sudo snap go --classic` on AWS Ubuntu). Then:

   ```bash
   mkdir client-demo && cd client-demo
   go mod init client-demo
   go get github.com/gravitational/teleport/api/client
   ```
   
5. Create a file `main.go` with the following command, modifying the `Addrs` strings as needed:

   ```bash
   cat > main.go << 'EOF'
   package main

   import (
      "context"
      "log"
      "github.com/gravitational/teleport/api/client"
   )

   func main() {
      ctx := context.Background()

      clt, err := client.New(ctx, client.Config{
         Addrs: []string{
            "tele.example.com:443",
            "tele.example.com:3025",
            "tele.example.com:3024",
            "tele.example.com:3080",
        },
         Credentials: []client.Credentials{
            client.LoadProfile("", ""),
         },
      })

      if err != nil {
         log.Fatalf("Failed to create client: %v", err)
      }

      defer clt.Close()

      resp, err := clt.Ping(ctx)

      if err != nil {
         log.Fatalf("Failed to ping server: %v", err)
      }

      log.Printf("Example success!")
      log.Printf("Server response: %s", resp)
      log.Printf("Server version: %s", resp.ServerVersion)
   }
   EOF
   ```

6. Execute: `go run main.go`. On success:

   ```bash
   2021/05/06 17:37:09 Example success!
   2021/05/06 17:37:09 Server response: {tele.go-teleport.com 6.1.5 Kubernetes:true App:true DB:true  tele.go-teleport.com:443 {}  %!s(int32=0)}
   2021/05/06 17:37:09 Server version: 6.1.5  
   ```

   Note the client `resp` object which contains configuration information.

Review the `client` [pkg.go reference documentation](https://pkg.go.dev/github.com/gravitational/teleport/api/client) for more information about working with the Teleport API programmatically.
